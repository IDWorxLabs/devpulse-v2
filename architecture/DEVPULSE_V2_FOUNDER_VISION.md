## Authority Statement

This document is the **highest-level strategic authority** in DevPulse V2. It defines identity, mission, purpose, and permanent principles.

Architectural law (`DEVPULSE_V2_CONSTITUTION.md` and specialized law documents) governs **how** DevPulse is built safely. This document governs **why** DevPulse exists and **what** it must become.

When strategy and implementation conflict, strategy defines the destination; architecture defines the safe path to reach it. Neither may be overridden by feature code, automated systems, or intelligence layers without founder amendment.

## 1. DevPulse Identity

### What DevPulse Is

**DevPulse is an AI-powered software creation operating system.**

It is the environment in which ideas become functioning software — not through disconnected tools, but through a unified system that observes, understands, plans, builds, verifies, repairs, and learns across the full lifecycle of software creation.

DevPulse treats software creation as an **operating problem**, not a chat problem. The conversation is the control surface; the work is real.

### What DevPulse Is Not

| DevPulse is NOT | Why |
|-----------------|-----|
| A chatbot | Chat is the interface, not the product |
| Merely a coding assistant | DevPulse orchestrates the full creation lifecycle |
| A passive Q&A system | DevPulse acts, verifies, and reports on action |
| A dashboard of disconnected panels | Complexity is accessed through conversation |
| A black box that claims success | DevPulse shows its work and proves its claims |
| A validator that ignores browser reality | What the user experiences is the truth |

### What DevPulse Becomes in Practice

DevPulse is simultaneously:

| Role | Function |
|------|----------|
| **Builder** | Transforms plans into working software |
| **Verifier** | Proves that software works as claimed |
| **Observer** | Watches execution, UI, runtime, and system state |
| **Planner** | Structures ideas into architecture and execution plans |
| **Teacher** | Explains what it is doing and why |
| **Diagnostician** | Detects failures, drift, and hidden problems |
| **Recovery system** | Repairs, rolls back, and restores when things break |
| **Intelligence platform** | Unified reasoning layer shared across all capabilities |

These roles are not separate products. They are facets of one operating system with one source of truth and one answer authority.

## 2. DevPulse Mission

DevPulse exists to **close the gap between intention and working software** — with visibility, trust, and speed that ordinary AI tools do not provide.

### Mission Statements

1. **Reduce software creation friction** — Remove the overhead between "I have an idea" and "it works in the browser."

2. **Increase visibility into execution** — Users always know what DevPulse is doing, why, and where it is in the process.

3. **Prevent hidden failures** — No silent errors, no validator-green/browser-red divergence, no invisible broken paths.

4. **Provide trustworthy verification** — Claims are backed by evidence, browser reality, and explicit verification — not assertion.

5. **Accelerate idea-to-product** — Help users move from concept to functioning product faster than traditional development workflows.

6. **Make DevPulse understandable** — Users should never wonder what DevPulse is doing or why it made a decision.

### Mission in One Sentence

> DevPulse turns ideas into verified, working software while showing its work every step of the way.

## 3. DevPulse Purpose

DevPulse operates through a continuous cycle. This cycle is the purpose of every subsystem DevPulse will ever contain.

```
    ┌──────────┐
    │ OBSERVE  │ ← Watch reality: browser, runtime, UI, logs, state
    └────┬─────┘
         ▼
    ┌──────────┐
    │UNDERSTAND│ ← Interpret what observation means in project context
    └────┬─────┘
         ▼
    ┌──────────┐
    │   PLAN   │ ← Propose architecture, steps, risks, and verification
    └────┬─────┘
         ▼
    ┌──────────┐
    │  BUILD   │ ← Generate, modify, and orchestrate software creation
    └────┬─────┘
         ▼
    ┌──────────┐
    │   TEST   │ ← Exercise the build against expected behavior
    └────┬─────┘
         ▼
    ┌──────────┐
    │  VERIFY  │ ← Prove correctness with evidence, not claims
    └────┬─────┘
         ▼
    ┌──────────┐
    │  REPAIR  │ ← Fix failures, rollback bad changes, recover state
    └────┬─────┘
         ▼
    ┌──────────┐
    │  LEARN   │ ← Capture what worked, what failed, and why
    └────┬─────┘
         ▼
    ┌──────────┐
    │ IMPROVE  │ ← Apply learning to future plans and execution
    └────┬─────┘
         │
         └──────→ REPEAT
```

Every future DevPulse capability — Central Brain, AiDev Engine, Trust Engine, Project Vault, Self Vision, Reality Replay, World 2 — must serve one or more steps in this cycle. Capabilities that do not serve the cycle do not belong in DevPulse.

## 4. Core Product Principles

These principles are permanent. They apply from Phase 1 foundation through final-state maturity.

### P-1 — Architecture Before Features

Structure and law precede capability. DevPulse V2 was founded on constitutional architecture because V1 proved that features without guardrails destroy the product they intend to build.

**Implication:** No feature ships without ownership registration, budget compliance, and browser verification.

### P-2 — Browser Reality Above Validator Reality

What the user experiences in the browser is the supreme truth. Internal validators, headless tests, and Node VM traces are advisory — never authoritative over browser failure.

**Implication:** A system that passes validators but fails the browser is broken, regardless of report status.

### P-3 — One Answer Authority

Every user question has exactly one authoritative answer path. No competing intelligence layers, no post-processing overrides, no load-order-dependent outcomes.

**Implication:** Users receive one honest answer from one known source — not whichever subsystem loaded first.

### P-4 — One Source of Truth

Each domain of state, memory, history, and verification has one write owner and one authoritative store.

**Implication:** No duplicate timelines, no clobbered globals, no parallel reality systems.

### P-5 — Visible Execution

DevPulse never hides its work. Users see what is happening, why, and what stage the process is in — inline, in conversation, without hunting through panels.

**Implication:** The Operator Feed is not optional decoration; it is core product surface.

### P-6 — Human Control

DevPulse proposes, explains, and verifies. Humans approve consequential actions. Autonomous execution exists only within governed boundaries (World 2) with explicit founder authority.

**Implication:** No silent apply, no hidden mutation, no autonomous override of constitutional or vision law.

### P-7 — Verification Before Trust

Trust is earned through evidence, not claimed through confident language. DevPulse verifies before it asserts; it shows proof before it asks for belief.

**Implication:** The Trust Engine (evolution of UVL) is a verification authority, not a report generator.

### P-8 — Diagnostics Must Not Block Productivity

Observation and diagnosis serve the user — they never hold the user hostage. Chat, building, and interaction continue even when diagnostics fail or are incomplete.

**Implication:** Diagnostics run in background tiers; they never gate startup, answers, or clickability.

### P-9 — Chat-First Interaction

Chat is the primary DevPulse interface. Complex systems remain accessible through conversation. Users should not need to navigate a maze of panels to control DevPulse.

**Implication:** Every capability must be reachable and understandable through chat; secondary surfaces supplement, never replace.

## 6. DevPulse Success Definition

DevPulse succeeds when a founder or developer can:

### Immediate Success (Phase 1 Foundation)

1. Open DevPulse and interact within 2 seconds
2. Describe an intent in chat and receive an honest answer within 3 seconds
3. See what DevPulse is doing inline — not in a hidden panel
4. Trust that the answer came from one known authority
5. Continue working even if background diagnostics fail

### Mature Success (Final State)

1. **Describe an idea** in natural language and receive an improved, structured concept
2. **Review a proposed architecture** with clear rationale and risk assessment
3. **Watch DevPulse build** with visible execution at every stage
4. **See verification evidence** before being asked to trust any claim
5. **Recover from failures** through replay, diagnosis, and governed repair
6. **Understand every decision** DevPulse made — what, why, and what alternatives existed
7. **Move from idea to working product** faster than traditional development — without sacrificing quality or visibility
8. **Learn from DevPulse** as it learns from execution — capability grows through governed loops

### Success Metrics (Qualitative)

| Dimension | Success looks like |
|-----------|-------------------|
| Speed | Fast to open, fast to answer, fast to act — never frozen |
| Trust | Every claim backed by visible evidence |
| Visibility | Zero invisible execution; Operator Feed always honest |
| Intelligence | One coherent brain, not competing subsystems |
| Control | Human in command; DevPulse in service |
| Recovery | Failures are diagnosable, replayable, and repairable |
| Growth | System learns without drifting from mission |

### Failure Definition

DevPulse has failed if:

- Users wait more than a few seconds to interact without explanation
- Validators pass while browser reality fails
- Users cannot see what DevPulse is doing during execution
- Multiple systems give conflicting answers to the same question
- Diagnostics block chat, building, or interaction
- DevPulse claims success without verification evidence
- Mission drift occurs — DevPulse becomes "just another AI assistant"
