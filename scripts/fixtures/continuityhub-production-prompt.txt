Build ContinuityHub — a production Continuity of Operations (COOP) and incident-command platform for multi-site organizations.

PRODUCT IDENTITY
ContinuityHub is not a CRM, not a generic task tracker, and not a project management suite. It is an operations continuity console. Stakeholders appear as contacts only in the sense of notified parties. "Inventory" means failover capacity / spare kits / cold-spare site kits — not retail stock. "Timeline" means the incident command timeline — not a Gantt chart.

CORE CAPABILITIES (must all be present)
1. Incident command board — open incidents, severity, commander, status, linked sites.
2. Incident timeline — ordered operational events, decisions, and evidence attachments.
3. Stakeholder directory — roles (commander, PIO, legal, exec sponsor), notification preferences.
4. Runbook library — linked to incident types, with step checklists and ownership.
5. Escalation paths — severity → role chain, SLAs, acknowledgement tracking.
6. Dependency maps — critical systems, vendors, upstream/downstream failure impact.
7. Status page publisher — internal/external messages with approval gates.
8. Post-incident review workspace — timeline import, contributing factors, actions.
9. Evidence locker — immutable pointers to logs, screenshots, tickets (metadata only).
10. On-call rotations — schedules, handoff notes, coverage gaps.
11. Change windows — freeze periods, exception requests, conflict detection vs incidents.
12. Risk register — continuity risks, residual rating, owners, review cadence.
13. Communication templates — SMS/email/voice scripts parameterized by incident fields.
14. Audit trail — who changed severity, who published status, who closed incidents.

INFORMATION ARCHITECTURE
Primary navigation must be ContinuityHub product modules only (incident board, timeline, stakeholders, runbooks, escalations, dependencies, status publisher, PIR, evidence, on-call, change windows, risks, templates, audit). Do not inject banned fallback shells (projects, tasks, team, deals, leads, expenses) unless contract-bound by this prompt's exact continuity meaning.

DATA & WORKFLOWS
- Create incident → assign commander → auto-start timeline → propose escalation path.
- Acknowledge escalation steps; block severity downgrade without commander note.
- Attach evidence metadata; never invent binary blobs.
- Publish status only after approval when severity ≥ high.
- Close incident requires PIR draft link or explicit waiver.

NON-GOALS
No payments. No ecommerce inventory. No CRM pipeline. No generic todo list. No HarborFlow / ExpeditionOS / ResilienceOS branding.

QUALITY BAR
Production-faithful modules, local persistence, preview-ready navigation, validation gates, and contract traceability from this ContinuityHub prompt into generated feature modules.
